## はじめに

Claude Code の Skills は、プロンプトをパッケージ化して再利用する仕組みだ。フロントエンドのデザイン生成やコードレビューに使うのが一般的だが、**コードを書かないタスク**にも応用できる。

この記事では、「個人開発者向けに、最新技術トレンドから穴場のビジネスアイデアを発掘する」スキルを自作し、実際に実行した結果を紹介する。

やったことは2つ。

1. **スキルの設計と作成** — WebSearchを活用した4フェーズのリサーチ構造、5軸評価フレームワーク、参照ドキュメント群の設計
2. **スキルの実行** — フルスキャンモードで実行し、2026年2月時点の3つの具体的なプロダクト提案を得た

## スキルの全体像 — 何を作ったか

作成したスキルは `indie-idea-scout`。以下のファイル構成で `~/.claude/skills/` に配置した。

```
indie-idea-scout/
├── SKILL.md                           # スキル定義（4フェーズの実行手順）
└── references/
    ├── evaluation-framework.md        # 5軸評価のルーブリック
    ├── market-analysis-patterns.md    # 個人開発の成功パターン6種
    └── search-strategy.md             # WebSearch戦略ガイド
```

`/indie-idea-scout` と入力するだけで、以下が自動実行される。

```
Phase 1: トレンドリサーチ（WebSearch 10-14回）
    ↓
Phase 2: ギャップ分析（3種類のギャップを特定）
    ↓
Phase 3: アイデア生成・5軸評価
    ↓
Phase 4: 構造化レポート出力
```

## 設計のポイント — スキルに「分析力」を持たせる

### SKILL.md は「実行手順」に徹する

SKILL.md の frontmatter はシンプルに保つ。

```yaml:SKILL.md
---
name: indie-idea-scout
description: >
  個人開発者向けのニッチな収益化アイデアを発掘するスキル。
  WebSearchで最新のテック動向・API・規制変更を調査し、
  大企業が参入していない隙間市場を特定して、
  具体的なB2Cプロダクト提案を構造化レポートとして出力する。
---
```

本文には4つのフェーズの実行手順と出力フォーマットを書く。ポイントは**「何を」ではなく「どう調べるか」を指示すること**。

たとえば Phase 1 のトレンドリサーチでは、4つの調査カテゴリを明示した。

- **新API・プラットフォーム変更** — 直近で公開されたAPI
- **消費者の不満** — Reddit等でのペインポイント
- **成長中のニッチ市場** — Indie Hacker事例
- **規制・社会変化** — 法改正が生む新需要

最初はカテゴリを指定せず「最新のトレンドを調べて」とだけ書いていた。すると Claude は AI 関連のニュースを3-4件拾って終わり、という結果になった。**カテゴリを明示したことで検索が10回以上に増え、消費者の不満や規制変化といった「技術ニュース以外の角度」からもギャップを見つけるようになった**。

### references/ で「判断基準」を分離する

スキルの品質を左右するのが `references/` ディレクトリだ。今回は3つの参照ファイルを作った。

| ファイル | 役割 |
|---------|------|
| evaluation-framework.md | 5軸スコアリングの採点基準 + 除外/警告フィルタ |
| market-analysis-patterns.md | 個人開発で成功しやすい6パターンの定義 |
| search-strategy.md | WebSearchのクエリテンプレート集 |

なぜ SKILL.md に全部書かないのか。**プロンプトが肥大化すると、Claude は全体を均一に参照せず、後半の指示を軽視する傾向がある**。詳細な基準は別ファイルに分離し、SKILL.md からは「`references/evaluation-framework.md` を参照」と一行で指示する方が、確実に読み込まれる。

### 試行錯誤 — 最初のバージョンとの違い

最初に作ったバージョンは SKILL.md だけで references/ がなかった。結果はこうだった。

```diff
- 最初のバージョン（references/ なし）:
  「AIを使った画像編集ツールが面白そうです」
  「チャットボットのカスタマイズ需要があります」
  → 抽象的。誰でも思いつく。スコアもなく優先順位が不明

+ 改善後（references/ 3ファイル追加）:
  「Seedance 2.0 APIが2/10に公開。ECセラー特化の動画生成は未登場」
  「評価: タイミング5, 実現可能性5, 総合4.4/5.0」
  → 具体的。根拠URL付き。スコアで優先順位が明確
```

違いを生んだのは主に2つのファイルだった。

**評価フレームワークの効果**: 5軸評価を入れる前は「面白いアイデア」が出ていた。入れた後は「面白い、かつ個人開発者が今すぐ着手できて収益化できるアイデア」に変わった。特に「個人開発適性」軸が効いた。この軸がないと、チームが必要なアイデアや運用負荷が高いアイデアも混ざってくる。

**検索戦略ガイドの効果**: クエリテンプレートがないと Claude は英語で1-2回検索して終わる。テンプレートを与えると「英語でも日本語でも検索しよう」「Redditでペインポイントも調べよう」と動き、検索回数が3-4回から10回以上に増えた。情報量の差がそのままアイデアの質に直結する。

## 実行結果 — 何が出てきたか

スキルを実行した結果、2026年2月時点で3つの具体的なプロダクト提案が得られた。ここからは実際のレポートを要約して紹介する。

### 検出されたトレンド

WebSearchで見つかった注目すべき動き。

- **Seedance 2.0 API 公開（ByteDance）**: 2月10日リリース。AI動画生成APIが $0.10-$0.80/分の破格で一般開発者に開放
- **Gemini 3 Computer Use API**: Google がAIによるブラウザ自動操作をプレビュー提供
- **米国プライバシー規制の波**: 複数州で新プライバシー法施行。コロラドAI法が2026年6月に施行予定
- **AI一貫キャラクター生成の成熟**: 複数シーンでキャラの一貫性を保つ技術が実用段階に

### アイデア1: ClipForge — 商品ショート動画ジェネレーター（総合4.4/5.0）

商品写真をアップロードするだけで、TikTok / Instagram Reels 向けのショート動画を自動生成するWebアプリ。

| 評価軸 | スコア |
|--------|-------|
| タイミング | 5 — Seedance API 公開直後。EC向け動画ツールは未登場 |
| 実現可能性 | 5 — OpenAI互換APIでラッパー開発は容易 |
| 収益性 | 4 — 月額¥980-1,980。100ユーザーで月12万円 |
| 競合優位性 | 4 — 「ECセラー特化」は未開拓 |
| 個人開発適性 | 4 — APIラッパー型でサポートコスト低 |

**パターン判定**: APIラッパー型。タイムウィンドウは1-3ヶ月（API公開直後が勝負）。

**Why Now**: Seedance 2.0 が2月10日にリリースされ、APIが2月下旬に一般公開。画像→動画変換が$0.10/分から使える。メルカリやBASEの個人セラーは「動画がないと売れない」と感じているが、動画編集のスキルがない。

### アイデア2: KomaStudio — マンガ/Webtoon AI キャラクターエンジン（総合4.0/5.0）

AIでマンガのキャラクターを一貫した画風で複数コマに生成できるツール。キャラクターシートを一度作れば、様々なポーズ・表情を即座に生成。

| 評価軸 | スコア |
|--------|-------|
| タイミング | 4 — キャラクター一貫性技術が成熟期 |
| 実現可能性 | 3 — LoRA/IP-Adapterの学習コストあり |
| 収益性 | 5 — 月額¥1,480-2,980。100ユーザーで月18万円 |
| 競合優位性 | 5 — マンガ/Webtoon特化の直接競合なし |
| 個人開発適性 | 3 — 画像生成のインフラ管理が必要 |

**パターン判定**: コミュニティ特化型。タイムウィンドウは長い（6-12ヶ月）。

**Why Now**: Neolemon、LTX Studio 等のキャラクター一貫性ツールが登場したが、全て西洋の画風（Pixar風）中心。日本のマンガ・Webtoon画風に特化したツールは空白地帯。

### アイデア3: PriceHound — AI搭載 Web変更監視（総合3.6/5.0）

Webサイトの変更を監視し、AIが「価格が¥2,980→¥1,980に値下がりしました」のように要約して LINE に通知するツール。

| 評価軸 | スコア |
|--------|-------|
| タイミング | 3 — Gemini 3 Flash APIは新しいが、監視ツール自体は既存カテゴリ |
| 実現可能性 | 4 — 技術的に既知のパターン |
| 収益性 | 3 — 月額¥500-1,200。スケールが必要 |
| 競合優位性 | 4 — AI要約 + LINE通知 + 日本語 + 低価格の組み合わせは未開拓 |
| 個人開発適性 | 4 — ジョブキューの運用のみ |

**パターン判定**: 価格破壊型。Visualping（$14/月）の1/3の価格で日本語特化。

## 自分の「リサーチスキル」を作るには

今回のスキルはアイデア発掘だったが、同じ構造で様々なリサーチスキルを作れる。設計時に押さえるべき原則を3つにまとめた。

### 原則1: SKILL.md には判断基準を書くな

SKILL.md に書くのは**ワークフロー（何をどの順番でやるか）** だけ。「良いアイデアとは何か」「信頼できるソースとは何か」といった判断基準は references/ に分離する。

理由は2つ。まず、SKILL.md が長くなると Claude が後半の指示を軽視する。次に、判断基準だけを差し替えれば別の用途に転用できる。たとえば evaluation-framework.md を「技術記事の品質基準」に差し替えれば、同じ4フェーズ構造で「技術記事のネタ探しスキル」が作れる。

### 原則2: 曖昧な検索をさせたくなければ、クエリの「型」を与える

WebSearchを使うスキルで最も差が出るのがここ。Claude に「最新トレンドを調べて」と言うと、1-2回の検索で満足する。**クエリテンプレートをカテゴリ別に用意する**と、検索回数と情報の多角性が劇的に変わる。

テンプレートを作るときのコツ。

- **英語と日本語の両方を用意する** — 英語圏の情報が先行していることが多い
- **ソースを指定する** — `site:reddit.com` や `site:news.ycombinator.com` で生の声を拾う
- **年号を必ず含める** — `{year}` をプレースホルダにして最新情報を取得させる

### 原則3: スコアリング基準を入れると「面白い」が「実行可能」に変わる

評価軸を与えない場合、Claude は「面白いけど実現困難」「面白いけど競合だらけ」なアイデアも出す。スコアリング基準を入れると、**各アイデアに点数と根拠が付く**。根拠が弱いアイデアは自然とスコアが低くなり、フィルタリングされる。

評価軸の設計で重要なのは、**自分の状況に合った軸を入れること**。今回は「個人開発適性」を入れたことで、チームが必要なアイデアが除外された。もし「副業でやりたい」なら「週末だけで運用できるか」軸を追加する、といった調整が効く。

### 応用例

同じ設計パターンで作れるスキルの例。

| スキル名 | Phase 1 | references/ |
|---------|---------|-------------|
| 技術記事ネタ探し | Zenn/Qiitaのトレンド + HackerNews検索 | 「バズる記事」の評価基準 |
| 競合分析 | 特定ジャンルの既存サービス網羅調査 | SWOT分析テンプレート |
| 技術選定リサーチ | フレームワーク/ライブラリの最新動向調査 | 選定基準（パフォーマンス、学習コスト等） |
| 登壇ネタ探し | カンファレンスのCFP + 直近の話題 | 「通るプロポーザル」の評価基準 |

共通するのは「**WebSearch で情報収集 → 独自の評価基準でフィルタリング → 構造化レポート出力**」という3ステップ。SKILL.md で収集カテゴリを決め、references/ に評価基準を置く。この型さえ押さえれば、コードを一行も書かずに Claude の分析能力を拡張できる。

## まとめ

Claude Code の Skills は、コード生成だけでなく**リサーチと分析**にも使える。

今回のスキル設計で得た知見をまとめる。

- **SKILL.md はワークフロー、references/ は判断基準** — この分離がスキルの再現性と品質を決める
- **Claude の検索は「型」で制御する** — カテゴリ分け + クエリテンプレートで網羅性が劇的に変わる
- **スコアリング基準が出力の質を左右する** — 「面白い」ではなく「実行可能で収益化できる」にフィルタリングされる
- **最初は雑でいい。references/ を足すたびに品質が上がる** — 一発で完成させようとせず、実行→改善のサイクルを回す

スキルのファイル構成は SKILL.md + references/ 3ファイルで合計約20KB。一度作れば何度でも実行でき、毎回最新のWebSearch結果に基づいた提案が得られる。

「こんなリサーチを毎回やっている」と感じたら、それがスキル化の候補だ。SKILL.md に手順を書き、references/ に判断基準を置く。Claude Code はそれだけで新しい分析能力を獲得する。
